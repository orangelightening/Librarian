## Current Capabilities

### Core Features

**Document Management**:
- Add/remove individual documents
- Sync directories recursively
- Track document changes via SHA-256 checksums
- Automatic re-chunking on updates
- `.librarianignore` for sensitive files

**Search & Retrieval**:
- Semantic search across all indexed documents
- Configurable result limits
- Citation tracking and source attribution
- AI-powered result aggregation

**Backend Support**:
- ChromaDB backend (default, custom chunking)
- Chonkie backend (intelligent semantic chunking)
- Runtime switching via environment variable
- Shared ChromaDB storage (data compatibility)

**CLI Tools** (secure file access):
- Execute whitelisted commands
- Read document contents
- List files in directories
- Literal text search within files
- Document structure summary

### Supported Operations

**Safe for Concurrent Reads**:
- ✅ `search_library` - Multiple models can search simultaneously
- ✅ `list_indexed_documents` - No conflicts
- ✅ `get_library_stats` - Thread-safe
- ✅ `get_document_status` - Read-only, safe

**Requires Single Writer**:
- ⚠️ `sync_documents` - Metadata store not thread-safe
- ⚠️ `add_document` - Potential race conditions
- ⚠️ `remove_document` - Write conflicts possible

### Performance Characteristics

**Chonkie Backend**:
- Chunking quality: Excellent (semantic boundaries)
- Processing speed: Fast (~0.5s per document)
- Search relevance: Better than custom chunking
- Memory usage: Moderate (Chonkie + torch dependencies)

**ChromaDB Backend**:
- Chunking quality: Good (fixed-size chunks)
- Processing speed: Very fast
- Search relevance: Good
- Memory usage: Low (minimal dependencies)

## Known Limitations

### Current Constraints

**Transport**:
- stdio only (local process communication)
- No network access
- Single client connection

**Concurrency**:
- Multi-model read: Safe ✅
- Multi-model write: Unsafe ❌
- No file locking in metadata store
- No thread-safe singleton initialization

**Security**:
- No authentication (relies on local execution)
- No encryption (stdio transport)
- CLI tools have file system access (sandboxed to SAFE_DIR)
- `.librarianignore` is only access control

**Scalability**:
- Designed for single-user workflows
- No user isolation or namespaces
- No audit logging
- No rate limiting

### Technical Debt

**Metadata Store** (`mcp_server/core/metadata_store.py`):
- No file locking (race condition risk)
- JSON-based (not ACID compliant)
- No concurrent write handling
- No migration/versioning system

**Singleton Pattern** (`mcp_server/tools/library_tools.py`):
- Module-level globals not thread-safe
- Lazy initialization without locks
- Could cause double-initialization under concurrency

**Error Handling**:
- Some exceptions caught and printed (not propagated)
- No retry logic for transient failures
- No circuit breakers for external dependencies

## Deployment Status

### Production Ready: Single User ✅

**Deployment Options**:
```bash
# Option 1: ChromaDB backend (default)
./setup_mcp.sh

# Option 2: Chonkie backend (recommended)
export LIBRARIAN_BACKEND=chonkie
./setup_mcp.sh
```

**MCP Client Configuration** (Jan, LM Studio, etc.):
```json
{
  "mcpServers": {
    "librarian": {
      "command": "/path/to/venv/bin/python",
      "args": ["/path/to/mcp_server/librarian_mcp.py", "--safe-dir", "/path/to/docs"],
      "env": {
        "PYTHONPATH": "/path/to/librarian-mcp",
        "LIBRARIAN_BACKEND": "chonkie"
      }
    }
  }
}
```

**Installation**:
```bash
# Clone repository
git clone <repo-url>
cd librarian-mcp

# Create virtual environment
python -m venv venv
source venv/bin/activate

# Install dependencies
pip install -r requirements.txt

# Run server
./setup_mcp.sh
```

**Dependencies**:
- fastmcp (MCP framework)
- chromadb>=0.4.0 (vector database)
- chonkie[all]>=1.6.0 (intelligent chunking)

**System Requirements**:
- Python 3.10+
- 500MB disk space (including torch for Chonkie)
- 2GB RAM recommended
